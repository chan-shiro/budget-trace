# データ収集戦略 — 多自治体展開の方針

個々の資料の取得手順は [data-sources.md](data-sources.md)、層構造の規約は [architecture.md](architecture.md) §10。
本書は「**どの順番で・どの種類のデータを・どうやってスケールさせるか**」の方針を定める。

## 大原則

1. **実データのみ**。一次資料が無い機能はダミーで見せず「未収録」とする（CLAUDE.md のデータ来歴ルール）
2. **1パーサの効果範囲を最大化する**。1,741自治体×個別対応は成立しない。
   全国統一資料を使い尽くしてから、様式の割れる資料に手を出す
3. **検証ゲートが LLM・手作業を安全にする**。予算・決算データは「合計＝内訳の和」で
   自己検証できるので、抽出手段が何であれ間違いは normalize/derive の前に弾ける。
   新しいデータ種別を足すときは**先に検証ルールを設計する**
4. 発見・手順・クセは必ず [data-sources.md](data-sources.md) に記録してから次へ進む

## 3層の攻め方

### 第1層: 全国統一データ（総務省・e-Stat）— 1パーサ = 1,741自治体

費用対効果が圧倒的。優先的に使い尽くす。

| 資料 | 得られるもの | 状況 |
| --- | --- | --- |
| 市町村別決算状況調（1）概況 | 人口・歳入歳出総額・面積・産業構造・財政力指数・経常収支比率・実質公債費比率・将来負担比率 | ✅ 収録済み |
| 同（3）目的別歳出内訳 | 款＋項レベルの歳出決算 | ✅ 収録済み |
| 同（2）歳入内訳 | 28科目＋内訳（ふるさと納税・普通/特別交付税・国庫支出金の内訳など） | ✅ 収録済み |
| 同（4）性質別歳出・（5）地方債 | 人件費/扶助費/普通建設…（主要14性質）、地方債現在高・積立金・債務負担行為 | ✅ 収録済み（**R2〜R6・全1,741市町村**・年度ドロップダウンで経年。別ソース `soumu-shichoson-seishitsu` に隔離） |
| 地方財政状況調査 個票（e-Stat） | 目的別×性質別クロス等、最深の全国統一データ | 未調査 |
| 財政状況資料集 | 類似団体比較の公式版・経年 | 未調査 |
| 決算カード（e-Stat） | 1自治体1枚の標準サマリ・経年 | 未調査 |

**ここまでで「どの自治体を選んでも決算ベースのダッシュボードが出る」状態を目指す。**

### 第2層: 様式が割れる自治体資料 — テンプレート化＋LLMフォールバック

予算資料・財政事情公表・補正予算書など。自治体ごとに様式が違うが、
数パターン（表型PDF・スライド型PDF・Excel/CSV公開型）に収斂する想定。

- 甲府で作ったパーサ（款別一覧・主な事業・財政事情）を**様式テンプレート**に一般化し、
  registry の `parserOptions`（ページ番号・列マップ）で自治体差を吸収する
- テキスト層のある PDF は決定的パース最優先（`pdftotext -layout` / `-tsv` の単語座標）
- テンプレートに乗らない資料だけ **LLM 抽出 → Zod 検証 → 整合チェック**。
  検証ゲート（合計＝内訳の和・前年比の再計算一致など）を通らない抽出は採用しない
- 東京都・千葉市など予算をオープンデータ（CSV）公開する自治体は個別に拾う（労力小・効果大）

### 第3層: 対象自治体の広げ方 — 「比較が濃くなる順」

全方位に広げず、画面価値が上がる順に:

1. **甲府の類似4市**（和泉・沼津・豊川・山口）— 類似自治体比較が予算ベースでも成立する
2. **山梨県内**（市区町村選択画面が「準備中」から埋まっていく）
3. 人口上位都市 → 全国

## クローラー方針（自動取得の設計）

現状は**クローラーではない**: 人間（またはエージェント）が registry に直リンクを登録し、
`pipeline:fetch` が単発取得するプル型。自動化を進めるときは以下の設計に従う。

### 発見は貪欲に、取得は行儀よく

**発見（discovery）と取得（fetch）を分けて考える。** 取得は下記のとおり控えめでよいが、
発見は貪欲でよい — ウェブ検索・検索エンジン・LLM を総動員して「どの自治体の・どの資料が・
どこにあるか」を先回りで洗い出し、data-sources.md の候補表と registry を育てる。

- **ウェブ検索の活用**: `site:city.○○.jp 当初予算` `○○市 財政事情` などの定型クエリで
  ランディングページを一括発見する。検索エンジンへの問い合わせは相手サイトへの負荷ではない
- **LLM 支援**: 検索結果・ページ構造から「予算資料PDF の直リンク」「公表サイクル」を抽出し、
  registry 登録候補（id・urls・landingPage・parserOptions の見込み）まで下ごしらえする
- **発見の成果は必ず記録**: 未登録でも data-sources.md の候補表に載せる（発見コストを二度払わない）
- 登録（＝取得対象化）の最終判断は人間。ライセンス・スコープ・パーサ適合を確認してから

### 原則（取得側） — 「巡回」ではなく「点検」

対象は政府・自治体サイトで、資料の更新は年1〜2回。サイトを再帰的にクロールする必要は
一切なく、**登録済み URL とランディングページの定期点検**だけで足りる。

- **再帰クロールはしない**。取得対象は registry に登録された URL のみ
- **頻度は公表サイクルに合わせる**（下表）。それ以外の高頻度アクセスはしない
- **同時接続は1**・失敗時は指数バックオフ・リトライは数回まで
- robots.txt を尊重する
- **UA の使い分け**: 手動・単発の fetch は現状どおりブラウザ相当 UA（DC IP 拒否対策）。
  定期実行するクローラーに移行する場合は、識別可能な UA（例:
  `budget-trace-bot/1.0 (+リポジトリURL)`）へ変更し、拒否されたら手動取得に戻す

### 3つの仕事

1. **更新検知（登録済み URL の点検）**: 定期 fetch → sha256 を raw-meta と比較。
   変化していたら新版としてアーカイブし、基準日（PDF 冒頭の「令和N年M月D日現在」等）を
   照合してから registry を更新する。同一 URL 上書き型資料（財政事情など）はこれが唯一の
   版管理手段（kofu-zaisei-jokyo で確立済みのパターン）
2. **新資料の発見（ランディングページの点検）**: landingPage の HTML からリンク一覧を取り、
   前回スナップショットとの差分（新しい年度ページ・新 PDF）を検出して**人間に提案する**。
   registry への登録は自動化しない（ライセンス・スコープ・パーサ適合の判断が要るため）
3. **通知**: 差分検知の結果は Issue / PR 提案の形で残す。黙って registry を書き換えない
4. **発見の4階層（消失前提の探索順序）**: 一次資料は発行元から消える（実例: 甲府市の
   R4・R5 予算資料、R3 決算状況ページ）。探索は次の順で行い、見つかった階層を registry に記録する:
   1. **現行サイト** — インデックスページ→年度ページ→資料。リンク切れでも Google の
      インデックスに残る旧 URL を検索して直叩きする（実例あり）
   2. **Wayback Machine** — CDX API（`web.archive.org/cdx/search/cdx?url=…`）で
      スナップショットを列挙。上書き型 URL は過去版の宝庫（財政事情 PDF の 2021/2024 年版）
   3. **WARP（国立国会図書館インターネット資料収集保存事業）** — 自治体サイトを
      **2005年から四半期ごと**に丸ごと保存している。甲府市は waid/4530。
      URL 形式は Wayback 同様の `warp.ndl.go.jp/web/<ts>/<元URL>`（ビューアシェル）と
      `warp.ndl.go.jp/<収集日>/<ts>/<元URL>`（pywb 実体 — 機械取得はこちら）。
      **Wayback に無くても WARP にはあることが多い**（実例: R3 収入支出詳細を
      2023-11-06 版から完全回収。市サイト削除・Wayback 未収録だった）。
      pywb はリンク書き換えを行うがテーブル等のテキストは原文のまま
   4. **情報公開請求・図書館** — ウェブに一度も載らなかった資料
      （予算書本編・主要な施策の成果報告書・事務事業評価の全詳細票など）の最終手段
5. **エビデンス3層コピーの原則（2026-07-13 正式化）**: 収録した一次資料は必ず**3つの写し**を持つ。

   | 層 | 実体 | 役割 |
   | --- | --- | --- |
   | ① git raw | `data/raw/`（コミット・sha256 固定） | 私たちの原本アーカイブ。パースの入力 |
   | ② Wayback | `bun run pipeline:archive` → `data/archives.json` 台帳 | **第三者が検証できる**中立な写し。`id_` URL で raw と sha256 突合（`sha256Match`） |
   | ③ 自サーバー配信 | `pipeline/sync-public-sources.ts` → `public/sources/`（dev/build 前段で自動） | 画面のエビデンス表示（ドロワー）。**PDF・HTML・Excel の全形式** |

   **リンクのルール**:
   - 画面のエビデンスリンクは**常に③（自サーバーの原本コピー）をドロワーで開く**。
     Wayback（②）と発行元（元 URL）はドロワー内の補助リンクに置く
   - **発行元・Wayback への直リンクを主リンクにしない** — 発行元の直リンクは中身だけ
     差し替えられ得る／外部アーカイブは遅く・消える可能性もゼロではない。
     コピーは収録時点の版に固定され、表示も速い
   - 例外: ランディングページ等の「ファイルでない」ナビゲーションは外部リンクで可
   - 新しい資料形式を扱うときは③の表示手段も揃える
     （PDF=PdfViewer / HTML=HtmlViewer サンドボックス / それ以外=ダウンロードカード）

   **運用**: 新資料は取得（①）したら**即座に② Wayback 登録**（発行元の削除は現実に起きる —
   甲府市 R4・R5 予算資料が実例。削除後は登録もできず、WARP に残っていたのは幸運）。
   ③は sync が自動。上書き型 URL は版が変わるたび `--force` で再登録。
   WARP 由来の URL は②の対象外（それ自体が国の恒久アーカイブ）

## 事業単位の執行情報の戦略（2026-07-12 調査）

「款別の執行」より深い**事業単位の執行×成果**を取るルート。甲府市での調査結果:

| 資料 | 中身 | 公開状況 | 収録戦略 |
| --- | --- | --- | --- |
| **行政評価（事務事業評価）** | 実施計画事業（主要約134）の評価一覧＋詳細票 | **H29〜R7 の全年度ページが現存**（/zaise/rNgyouseihyouka.html 等） | 一覧（xlsx/pdf）は決定的パース可能。事業名は予算資料の「主な事業」と照合できる |
| 同・詳細票（第2号様式） | **事業費の決算額3年分（財源内訳つき）・トータルコスト（人件費込み）・活動/成果指標の目標と実績** | 公開は**年数枚のサンプルのみ**（R7 Excel に2事業分など） | サンプルでパーサを確立 → **全事業分は情報公開請求**（電子データ指定で請求） |
| 主要な施策の成果報告書（自治法233条5項） | 決算添付の法定書類。施策・事業ごとの執行実績 | **ウェブ未掲載**（決算ページ・議会ページとも） | 情報公開請求 or 議会図書室。WARP の過去スナップショットも要走査（掲載時代があった可能性） |
| **統計書 財政章** | **款項×当初/最終/決算（円）** — 補正の規模と項レベル執行 | ✅ H30〜R6 の7年分収録（data-sources.md §6）。**統計ページは執行データの気づかれにくい宝庫** — 他自治体でも統計書を必ず偵察する | 収録済み |
| 教育委員会 点検・評価報告書 | 教育費事業の実績（法定・公開義務あり） | 未調査 | 次回偵察対象 |
| 補助金交付状況・入札/契約結果 | 事業の別断面（相手先・金額） | 未調査 | 次回偵察対象 |

→ 最短の一手: **行政評価の一覧＋公開詳細票の収録**（事業名×評価×コスト実績のサンプル）。
本命は情報公開請求による全詳細票（事業単位の決算額×成果指標が全量揃う）。

### リクエスト駆動の情報公開請求（2026-07-12 決定）

情報公開請求はコスト（手数料・写し実費・往復日数）がかかるため、**利用者のリクエストが
貯まったものから請求する**方式にする。台帳は GitHub Issues（バックエンド不要・全経緯が公開）:

1. **起票導線は2系統**: ①出典タブの「未収録の資料をリクエストする」一覧
   ②**データが無い箇所のその場チップ**（款ドリルの「事業掲載なし（詳細不明）」横、
   項別内訳が参考表示の年度、政策テーマの事業リストなど）。その場チップは
   `buildRequestUrl()`（data.ts）が **どの画面で・何が・なぜ欲しいか** を Issue フォームの
   フィールドにプリフィルする（GitHub issue form は URL クエリの id 一致でプリフィル可能）。
   Issue テンプレートは `.github/ISSUE_TEMPLATE/source-request.yml`、ラベル `資料リクエスト`
2. **賛同 = Issue の 👍 リアクション**（重複起票は Issue 側で统合）
3. **閾値（目安: 👍 5件）に達したら運営が情報公開請求**（電子データ（CSV/Excel 等）での
   交付を明示して請求する — 紙や画像 PDF だと再パースコストが跳ね上がる）。
   請求権者の範囲・手数料は甲府市情報公開条例を請求前に確認する
4. 開示資料は `pipeline:ingest`（手動投入）→ parse → validate → 収録 PR を Issue にリンクして close。
   **リクエスト→請求→開示→検証→収録の全経緯がトレース可能**（本サービスの思想と同型）
5. DB 移行後はネイティブなリクエスト/投票機能に置き換え（Issues が移行元台帳になる）

## 運用・基盤

- **registry の多自治体化**: source に muniCode を持たせ、自治体×資料のマトリクス台帳にする
- **更新の自動化**: 上記「クローラー方針」に従う（点検型・提案型。黙って書き換えない）
- **DB 化の判断（2026-07-12 整理）**: 現状の JSON + git を当面維持する。理由:
  - **横断比較のために DB は要らない** — 比較可能性はスキーマではなく分類の問題で、
    決算系は総務省の標準科目・団体コードで既に解決済み（normalized 層）。
    規模も 1,741行×数年度で、ファイルスキャンで十分
  - **「綺麗に正規化できるか」は正しい懸念で、答えは「全部は正規化しない」** —
    比較するのは標準分類に乗る軸（款・項・性質別・団体コード・年度）だけ。
    事業名・独自科目など自治体固有のものは正規化せず per-自治体のまま提供する
    （budget-book が normalized を通らない現行設計そのもの）
  - **git がファイルであることの価値**: 来歴の diff・監査・PR ベースの貢献受け入れ
  - **DB に切り替えるトリガー**（いずれかが来たら）: ①クローラー運用の状態管理
    （発見キュー・点検ジョブ・差分履歴）②アプリの API 化・全文検索（事業内容の横断検索）
    ③深掘り自治体×多年度で gen コミット方式が肥大（目安: 事実行 10万超）。
    移行先は architecture.md のとおり Postgres（parsed/normalized → テーブル、gen → API）。
    アドホック分析には移行前でも DuckDB で JSON を直接クエリできる
  - **運用形態の懸念（2026-07-12 追記）**: クローリングはローカルマシンで実施し、
    ウェブは情報提供（読み取り）のみにしたい。DB 化するとローカルの収集処理が
    本番 DB へ直接書き込む形になり、書き込み経路の管理が発生する。
    移行時は「ローカルで収集・検証 → 成果物を artifact（dump / JSON）として
    デプロイに同梱 or 一方向 import」の書き込み分離パターンを検討する
    （本番 DB は読み取り専用レプリカ相当に保つ）。現時点の結論: **当面 git 継続**
- **貢献の受け入れ**: data-sources.md のチェックリストに沿った「資料URL登録のPR」を
  受けられる形にしておく（登録→fetch→パース→検証は既存基盤で回る）

## カバレッジ階層（full / decision）と配信モデル（2026-07-13）

アプリは自治体を3階層で扱う。**「横に広げる（decision）」と「縦に掘る（budget→full）」は別物として管理する**:

| 階層 | 対象 | 中身 | データ源 |
| --- | --- | --- | --- |
| **decision** | **全1,741市町村** | 款別歳出（→項）・歳入科目（→内訳）・1人あたり・財政指標・類似比較・決算経年（R2〜R6） | 総務省 決算状況調（追加取得0） |
| **budget** | 類似4市（豊川・山口・沼津・和泉） | 上記＋**当初予算の款別＋前年当初比較**（1年度分） | 各市の当初予算資料 PDF（`munibudgets.gen.ts`） |
| **full** | 甲府市のみ | budget＋補正・主な事業・執行状況・事務事業評価（複数年度） | 甲府市の予算資料群 |

- 判定は団体コード（`FULL_MUNIS`／`BUDGET_MUNIS`、`decision-index.gen.ts`／`munibudgets.gen.ts`）。`D.tierOf(muniCode)` が `'full'|'budget'|'decision'` を返す。
- budget 階層は静的 gen（4市×1年で小さい）に焼き込み（decision のようなフェッチはしない）。主な事業/執行/評価はリクエストチップ。
- decision の画面で full 専用機能（主な事業・執行など）は出さず、**その場リクエストチップ**にする。
- **配信は都道府県シャード**: `public/decision/<県コード2桁>.json`（47ファイル・各年度の款/項・科目/内訳を内包・計〜30MB）を
  **自治体選択時にフェッチ**（`useDecisionData` フックが県単位でキャッシュ）。全断面（〜30MB）は
  クライアントに焼き込めないための構成。索引・出典メタ・PREF_CODES は `decision-index.gen.ts`（数KB・焼き込み）。
- 生成は `pipeline/derive-app-data.ts` の `deriveDecisionShards`（各自治体・各年度で Σ款=歳出総額 を自己検証してから書く）。
  表示ビルダは `src/client/lib/decision.ts`（`buildDecisionView`）。
