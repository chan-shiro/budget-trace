# データ収集戦略 — 多自治体展開の方針

個々の資料の取得手順は [data-sources.md](data-sources.md)、層構造の規約は [architecture.md](architecture.md) §10。
本書は「**どの順番で・どの種類のデータを・どうやってスケールさせるか**」の方針を定める。

## 大原則

1. **実データのみ**。一次資料が無い機能はダミーで見せず「未収録」とする（CLAUDE.md のデータ来歴ルール）
2. **1パーサの効果範囲を最大化する**。1,741自治体×個別対応は成立しない。
   全国統一資料を使い尽くしてから、様式の割れる資料に手を出す
3. **検証ゲートが LLM・手作業を安全にする**。予算・決算データは「合計＝内訳の和」で
   自己検証できるので、抽出手段が何であれ間違いは normalize/derive の前に弾ける。
   新しいデータ種別を足すときは**先に検証ルールを設計する**
4. 発見・手順・クセは必ず [data-sources.md](data-sources.md) に記録してから次へ進む

## 3層の攻め方

### 第1層: 全国統一データ（総務省・e-Stat）— 1パーサ = 1,741自治体

費用対効果が圧倒的。優先的に使い尽くす。

| 資料 | 得られるもの | 状況 |
| --- | --- | --- |
| 市町村別決算状況調（1）概況 | 人口・歳入歳出総額・面積・産業構造・財政力指数・経常収支比率・実質公債費比率・将来負担比率 | ✅ 収録済み |
| 同（3）目的別歳出内訳 | 款＋項レベルの歳出決算 | ✅ 収録済み |
| 同（2）歳入内訳 | 28科目＋内訳（ふるさと納税・普通/特別交付税・国庫支出金の内訳など） | ✅ 収録済み |
| 同（4）性質別歳出・（5）地方債 | 人件費/扶助費/普通建設…、地方債現在高 | 未収録（同じパーサ基盤で拡張可能） |
| 地方財政状況調査 個票（e-Stat） | 目的別×性質別クロス等、最深の全国統一データ | 未調査 |
| 財政状況資料集 | 類似団体比較の公式版・経年 | 未調査 |
| 決算カード（e-Stat） | 1自治体1枚の標準サマリ・経年 | 未調査 |

**ここまでで「どの自治体を選んでも決算ベースのダッシュボードが出る」状態を目指す。**

### 第2層: 様式が割れる自治体資料 — テンプレート化＋LLMフォールバック

予算資料・財政事情公表・補正予算書など。自治体ごとに様式が違うが、
数パターン（表型PDF・スライド型PDF・Excel/CSV公開型）に収斂する想定。

- 甲府で作ったパーサ（款別一覧・主な事業・財政事情）を**様式テンプレート**に一般化し、
  registry の `parserOptions`（ページ番号・列マップ）で自治体差を吸収する
- テキスト層のある PDF は決定的パース最優先（`pdftotext -layout` / `-tsv` の単語座標）
- テンプレートに乗らない資料だけ **LLM 抽出 → Zod 検証 → 整合チェック**。
  検証ゲート（合計＝内訳の和・前年比の再計算一致など）を通らない抽出は採用しない
- 東京都・千葉市など予算をオープンデータ（CSV）公開する自治体は個別に拾う（労力小・効果大）

### 第3層: 対象自治体の広げ方 — 「比較が濃くなる順」

全方位に広げず、画面価値が上がる順に:

1. **甲府の類似4市**（和泉・沼津・豊川・山口）— 類似自治体比較が予算ベースでも成立する
2. **山梨県内**（市区町村選択画面が「準備中」から埋まっていく）
3. 人口上位都市 → 全国

## 運用・基盤

- **registry の多自治体化**: source に muniCode を持たせ、自治体×資料のマトリクス台帳にする
- **更新の自動化**: 公表サイクルが分かっている資料（財政事情=年2回・同URL上書き、
  決算状況調=年1回）はスケジュール fetch＋ハッシュ差分検知。上書きURL資料は
  取得時に基準日を必ず照合する（kofu-zaisei-jokyo で確立したパターン）
- **サーバー層への移行タイミング**: 深掘り対象が数十自治体×複数資料になったら
  `*.gen.ts` 方式が破綻する。第3層フェーズ1〜2の間に architecture.md のサーバー層計画
  （parsed/normalized → Postgres、gen → API）へ乗り換える
- **貢献の受け入れ**: data-sources.md のチェックリストに沿った「資料URL登録のPR」を
  受けられる形にしておく（登録→fetch→パース→検証は既存基盤で回る）

## 現在地とネクストアクション

- 第1層: 決算状況調3種を収録済み。次は（4）性質別歳出 or e-Stat 個票の調査
- 第2層: 甲府で3パーサ確立（予算資料・主な事業・財政事情）。次は類似4市の予算資料で
  様式のバラつきを偵察し、テンプレート化の設計材料にする
- 第3層: 未着手（第2層の偵察結果待ち）
