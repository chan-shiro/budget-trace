# データソース台帳 — 一次資料の取得方法

収録済みの一次資料について「どこで見つけ、どう取得し、どう検証したか」を記録する。
機械可読な台帳は [pipeline/registry/sources.ts](../pipeline/registry/sources.ts)（URL・パーサ・ライセンス）。
本書はその背後にある**手順と発見の経緯**（再取得・年度更新のときに必要な知識）を残す。
多自治体への展開方針は [data-strategy.md](data-strategy.md)。

レイヤ構成（メダリオン対応）と共通規約は [architecture.md](architecture.md) §10。

---

## 新しい資料を追加するときのチェックリスト

1. **ランディングページを特定する**（検索は `site:city.xxx.jp` や `site:soumu.go.jp` を併用）。
   直リンクではなくランディングページを `landingPage` に記録する — 直リンクは構成変更で切れる
2. **直リンクを確認する**: `curl -sL <url> | file -` などで実体（PDF/Excel）を確認。
   政府系はブラウザ相当の User-Agent が要ることがある（fetch.ts は対応済み）
3. **利用条件（ライセンス）を調べる** — 資料と同じ調査で必ず押さえる。フッターの「利用規約」
   「著作権・リンクについて」→ オープンデータページ（政府標準利用規約準拠 / CC BY 表記）→
   資料の奥付、の順で探す。**発行元の表記を原文のまま** `license` に書く（区分 open /
   permission-required / unverified は `/coverage` の derive が原文から**自動判定**するので
   要約・意訳しない）。**利用条件ページの URL と確認日は本書の当該資料メモに残す**。
   迷ったら `unverified` のまま（安全側）。詳細は [data-strategy.md](data-strategy.md) の
   「資料と一緒に『利用条件』も必ず調べる」と「③の再配布ライセンスは『区分して開示する』」
4. **registry に登録**（id は `発行元-資料名-年度` の kebab-case）。ライセンス・スコープも記入
5. `bun run pipeline:fetch <id>` — raw 取得と同時に sha256・取得日時が raw-meta に固定される。
   **raw はコミットする**（原本アーカイブ。発行元の差し替え・削除に備える）
6. **パーサを書く前に構造を調べる**: Excel は `sheet_to_json(header:1)` で行列を目視、
   PDF は `pdftotext -layout`（表が崩れる場合は `-tsv` で単語座標）。
   **テキスト層のある PDF は決定的パースを最優先**し、LLM 抽出は最後の手段にする
7. **検証ルールを先に決める**: 予算・決算データは「合計 = 内訳の和」「歳入 = 歳出（予算）」
   「記載率 = 再計算率」など自己検証できる。error 1件でも normalize/derive に進まない
8. `bun run pipeline:archive <id>` — **Wayback Machine にも登録する**。
   raw の git アーカイブは「私たちの写し」でしかないので、第三者が検証できる中立な写し
   （魚拓）を残す。スナップショット URL は `data/archives.json`（台帳）に記録され、
   derive の `wayback()` が**画面のエビデンスリンクをコピー優先に差し替える**
   （直リンクは中身だけ差し替えられ得るため、収録時点の版に固定されたコピーの方が透明）。
   **発行元の削除は現実に起きる**（甲府市 R4・R5 当初予算資料は市サイトから削除済みで、
   Wayback にも無く入手不可になった実例）。取得したら即座に登録する。
   - **版の突合**: file のコピーは `id_` URL でバイト列を取得し raw と sha256 突合、
     結果を台帳の `sha256Match` に記録する。`false` は別版（古いスナップショット等）の印
     （実例: 財政事情 01ipankaikei.pdf の CDX 最新が 2024 年の旧版だった → `--force` で解消）
   - 直リンクが同じパスへ上書きされる資料（財政事情）は版が変わるたび `--force` で再登録
9. **自サーバー配信を確認する** — `pipeline/sync-public-sources.ts`（dev/build 前段で自動）が
   raw の PDF・HTML・Excel を `public/sources/` へ同期する。画面のエビデンスリンクは
   **常にこのコピー**をドロワーで開く（発行元・Wayback への直リンクを主リンクにしない —
   「エビデンス3層コピーの原則」は data-strategy.md）。新しい形式を追加したら
   ドロワー側の表示手段（PdfViewer / HtmlViewer / ダウンロードカード）も確認する
10. `parse → validate → normalize（全国比較データのみ）→ derive` を通し、
   本書に取得手順・ハマりどころを追記する

## 収録済みの資料

### 1. 総務省「市町村別決算状況調」（soumu-shichoson-kessan-r6）

全1,741市区町村の普通会計決算。全国比較レイヤ（normalized）の基盤。

- **見つけ方**: [決算状況調トップ](https://www.soumu.go.jp/iken/kessan_jokyo_2.html) → 年度ページ
  （例: `/iken/zaisei/r06_shichouson.html`）。ページは Shift_JIS なので grep する場合は
  `iconv -f SHIFT_JIS -t UTF-8` を通す
- **取るファイル**: 1年度 = **6ファイル**。「1 都市別」「2 町村別」それぞれの
  「（1）概況」（人口・歳入歳出総額・面積・産業構造・財政力指数等）、「（2）歳入内訳」
  （28科目＋内訳）、「（3）目的別歳出内訳」（款＋項）。（4）性質別・（5）地方債は未使用
- **歳入内訳のクセ**: 「地方税内訳」は「うち〜」の**部分列挙**（軽自動車税などが載らない）
  ため Σ内訳 ≠ 科目総額。検証は「うち〜を含む科目は Σ ≤ 総額、それ以外は厳密一致」。
  概況と歳入内訳の間に**原典側の1千円不整合**が存在し得る（R6 は伊賀市の1件、warning 扱い）
- **目的別歳出内訳には項レベルが入っている**: 款の内訳列（総務管理費・社会福祉費・
  児童福祉費・小学校費など約40列）をパース済み。直下の項は「款内訳/連番/項名」の3段、
  都市計画費・保健体育費は「連番+項名/(N)/目名」の4段ヘッダで目列を合算して項とする。
  検証は全自治体で「項の和＝款」の厳密一致
- **取得**: `bun run pipeline:fetch soumu-shichoson-kessan-r6`（直リンクは registry の `urls`）。
  クラウド環境からは 403 になることがある（DC IP 拒否）— ローカルなら通る。
  ダメなら手動DL → `pipeline:ingest`
- **構造のクセ**（パーサ `soumu-shichoson-kessan` が対応済み）:
  - ヘッダは複数行＋結合セル。「団体コード」を含む行をヘッダ先頭とし、結合セルは範囲展開して解決
  - **労働費・諸支出金は総額列がなく内訳列のみ** → 合算する
  - 末尾の「（参考）一般行政経費」ブロックが款名を再掲する → 列解決から除外
  - 都道府県名の列はない。「団体コード空・団体名だけ」の区切り行（**字間スペース入り**）で県が変わる
  - 特別区は都市別ファイルに含まれる。政令市の行政区は含まれない
- **検証**: 全自治体で「目的別合計 ≒ 歳出総額（±0.5%）」。R6 は 1,741自治体 error 0
- **過去年度も収録済み（R2〜R6 の5年分）**: 年度ごとに sourceId を分けて登録
  （soumu-shichoson-kessan-r2 〜 r6）。ファイル構成・ヘッダ形式は5年間同一で、
  同じパーサがそのまま通る。**年度による科目差**が2つ: R2 には「特別地方消費税交付金」が
  あり、「地方特例交付金」に「等」が付かない（R6 は「地方特例交付金等」）。
  REVENUE_COLUMNS に両表記を併記して吸収済み。検証は5年×1,741自治体で error 0
  （warning は R5・R6 の伊賀市 = 原典側の1千円不整合のみ）
- **健全化判断比率のクセ**: 実質公債費比率・将来負担比率は概況の「健全化判断比率」
  結合セルの2段目にある（h0 でなく h1 を見る）。「-」の団体は欠損として扱う
- **年度更新**: 年度ページから6ファイルの直リンク（`/main_content/00xxxxxxx.xlsx`）を確認して
  新しい sourceId（…-r7 など）で登録する

#### 1b. (4)性質別歳出・(5)地方債（soumu-shichoson-seishitsu-r6・2026-07-13 追加）

同じ決算状況調ページの (4)性質別歳出内訳・(5)地方債の4ファイル（都市別/町村別 × 4/5）。
**R2〜R6 の5年分収録済み**（年度ドロップダウンで性質別の経年が追える。R6=`001061672/673/677/678`、
R2〜R5 は各年度ページから直リンク確認）。**既存パーサ（概況/歳入/目的別）とは別ソース・別パーサ・
別 docType（`municipal-nature`）に隔離**して回帰リスクを断ち、decision シャードの derive で
**団体コードで年度ごとに結合**する（各年度スライスに `nature`/`bond` を付与）。

- **性質別の列特定**: ヘッダ行の**漢数字マーカー**（一=人件費 … 十四=前年度繰上充用金）が
  主要14性質の総額列にちょうど1つずつ立つ。この列の直下行を性質名として拾う
  （固定インデックスでなくマーカー基準なので年度差に強い）。`sheet_to_json` は
  **raw（merge展開なし）**で読む（merge展開すると漢数字が範囲全体に広がり誤検出する）
- **地方債の列**: ラベル一致（地方債現在高・積立金現在高・財政調整基金・減債基金・
  その他特定目的基金・債務負担行為額翌年度以降支出予定額・公営企業等に対する繰出金）
- **自己検証**: `natureTotal = Σ主要14性質`（parsed 内で厳密一致）。さらに derive/検証で
  **Σ性質 = 概況の歳出総額**を全1,741市町村で突合（R6 は完全一致・0件差）
- タイトルセルは先頭数行の空行後にあるので、ファイル種別判定は先頭20行を走査する
- 画面: decision ダッシュボードに「性質別の歳出」パネル＋「地方債・基金の状況」カード（R6のみ）

### 2. 甲府市「当初予算（案）資料」（kofu-yosansho-r8 / r7 / r6）

款別の歳入・歳出一覧（前年度比較つき）と「主な事業一覧」。アプリの款レベルと事業の原典。
**R2〜R8 の7年度分を収録済み**（年度ドロップダウンの原典）。R8〜R4 は同型の単一 PDF
（R5・R4 は WARP 回収）、R3・R2 は分冊形式。

- **見つけ方**: [甲府市 予算ページ](https://www.city.kofu.yamanashi.jp/shise/yosan/yosan/index.html)
  → 年度ページ（例: `/zaise/shise/yosan/yosan/r8yosan.html`）。
  「令和N年度甲府市当初予算資料（PDF）」が本体。**款項目節まで載った予算書「本編」は
  ウェブ未公開**（載っているのは款別一覧と主な事業のみ）
- **取得**: `bun run pipeline:fetch kofu-yosansho-r8`
- **構造のクセ**（パーサ `kofu-yosansho`）:
  - テキスト層あり。款別一覧は `pdftotext -layout` の行から数値トークン列で決定的に取れる
  - 主な事業一覧（p.14-23）は複数行セルの表 → `-tsv` の**単語座標**で列を判別。
    行は No. 列のY中点で分割、セクションは見出しフォント高（>=12pt）で追跡
  - 予算額と内容テキストが**1語に結合**されることがある（例 `5,850「ベビージュエリー」…`）→ 先頭数値を分離
  - 款セクションのほかに「介護保険事業特別会計」セクションがある（款パターン外）
  - ページ番号は registry の `parserOptions`（revenuePage / expenditurePage / projectPages）。
    資料構成が変わったらここを更新
- **検証**: 款の和＝資料記載合計（厳密一致）、歳入合計＝歳出合計、事業は No 連番・款予算超過なし。
  さらに**年度間クロスチェック**が効く: R8 の「前年度額」計 880.9億 = R7 総額、R7 の前年度額計 807.9億 = R6 総額（一致確認済み）
- **年度によるクセ**（パーサ対応済み）:
  - ページ番号が毎年違う（R8: 歳入p12/歳出p13/事業p14-23、R7: p13/p14/p15-25、R6: p15/p16/p17-25）→ registry の `parserOptions`
  - 主な事業の表ヘッダが R8「基本/目標」「施策」に対し R7・R6 は「基本/目標等」「施策等」
  - 基本目標列の左端 x 座標が揺れる（R8: ≈652 / R7・R6: ≈641）→ 列境界は 630
  - 基本目標の体系が違う: R8 は第七次総合計画（ひと/まち/魅力）、R7・R6 は第六次（基本目標1〜4・基本構想の推進）。
    「基本構想の推進」はセル内で2行に折り返される → パーサで連結を復元
  - 事業名の（ ）書き（予算書名）は**入れ子括弧**があり得る（例:（（仮称）〜整備事業費））→
    正規表現でなく末尾から括弧の対応を取って切り出す（R8 にも3件あり、修正済み）
- **R5・R4 は WARP から回収（収録済み）**: 年度ページと PDF が市サイトから削除済みで
  Wayback にも無かったが、**WARP の 2024-05-09 収集分に完全残存**していた（発見の4階層の実践例）。
  WARP 回収版のクセ:
  - **PDF の座標系が右寄り**（予算額 x≈314-330）→ `parserOptions.projectColumns` で列境界を上書き
  - **R5 に「上記のうち〜」の複合行**（1つの No に親事業＋内数の子事業。No.49 空家等対策）→
    パーサは予算額のY位置で行帯をサブ分割し、子行は `no: null`（内数）として収録
  - **R5 は市長改選年の骨格予算**（総額 790.4億）。R6 資料の前年列は6月補正の政策的予算を
    含む 799.4億で、R5 資料の総額と一致しない — R6 資料自体の注記を `prevNote` として
    parsed に取り込み、画面の前年比較に表示する（年度間クロスチェックの「不一致にも説明がある」例）
- **R3・R2 は分冊形式（収録済み）**: 「一般会計歳入歳出予算款別一覧表」と「主な事業」が
  別 PDF（registry の `parserOptions.kanFile / projectsFile` で割当）。クセ:
  - 款別一覧表は同型だが、**款名が2行に折り返し款番号が単独行になる**
    （株式等譲渡所得割交付金・交通安全対策特別交付金）→ パーサは断片を空行まで持ち越して結合
  - **R2 の前年列は「令和元年度 6月補正後予算額」**（当初でない）。パーサが
    「補正後予算額」の語で自動検出し `prevBasis: "補正後"` として parsed に記録。
    画面の前年比較・対前年度%はこの基準を明示する
  - **主な事業は箇条書き形式**（`projectFormat: "bullets"`）: 「● 事業名…N億M万円」
    （万円 → 千円 ×10）、★=新規/◆=繰越の補足行、「基本目標 N + 名称」「施策の柱」の
    章立て。款・掲載番号・予算書名の記載が無い（kan / no / budgetBookName は null）ため、
    款ドリルダウンには出ず、政策テーマ・注目の事業にのみ出る
  - 検証の年度間クロスチェック: R3 記載の前年度額 74,429,982千円 = R2 の総額（一致確認済み）
- **年度更新**: 年度ページの新 PDF を registry に登録。ページ番号を要再確認

款別の予算現額・収入/支出済額・収入率/執行率。「予算執行状況」タブの原典。

  版の同一性は raw アーカイブ＋sha256 で固定する。取得時に PDF 冒頭の
  「令和N年M月D日現在」で版を必ず確認し、registry の title / fiscalYear と一致させる
- **取得**: `bun run pipeline:fetch kofu-zaisei-jokyo-r7`

### パーサ一般化で吸収した自治体差（`parserOptions`）

- **合計行も小数フィルタ**が必要（構成比が金額間に入る豊川・山口・沼津）
- **判定順**: 合計行判定を見出しスキップより先に置く（見出し語「歳入」は「歳入合計」の部分文字列のため。

定型著作権表記で、著作権法32条2項（行政の広報資料は出典明示のうえ転載可。ただし禁止表示があれば不可）の
**「禁止する旨の表示」には当たらない疑い**がある。棚卸しの前に data-strategy.md の【未決】節を読むこと。

#### 政令市に共通する要注意（個別コメントでは繰り返さない）

- **款の体系が総務省の目的別ではない**。政令市の当初予算の款は**局ベースの独自体系**で、
  `民生費・衛生費・労働費・商工費` が存在せず、代わりに `こども未来費`（川崎）・
  `にぎわいスポーツ文化費`（横浜）・`保健福祉費`（札幌）・`こども育成費`（福岡）等が並ぶ。
  さらに**名古屋・札幌は人件費を全款から抜いた `職員費` を持つ**（札幌で歳出の13.3%）ため、
  他市と款別を直接比較すると保健福祉費等が過小に見える。
  **同じ市の decision 階層（総務省決算＝目的別）とは款名が食い違って見える**のは仕様。
  budget 階層は normalize を通さない（derive が parsed から直接構築）ので収録できる。
  山梨県の県固有款（警察費・県税）に先例あり。**標準款へ黙って寄せてはならない**。
- **廃止税目の行を款として拾えない**。款番号欄が `△`（札幌）・`▲`（福岡）・括弧書き（大阪）の
  廃止科目行があり、本年度 0 / 前年度のみ額を持つ。款番号が無いため `kanNo`（正の整数必須）で
  表現できず、**前年度 Σ が僅かに不足する**（札幌 694,000千円＝合計の0.05% / 福岡 1千円）。
  **validate は当年度 Σ しか検査しない**ので error にならず、画面の前年比較は資料記載の
  前年度合計を使うため表示上も矛盾しない（1.27兆の表示に6.9億は現れない）。
  **＝「静かに落ちている」状態**なので、ここに記録して既知とする。直すなら `kanNo` を
  nullable にする schema 変更が要る（0.05% に対しては過剰と判断）。

#### パーサ拡張（3点・2026-07-15）

1. **`revenuePages` / `expenditurePages`（ページ範囲）** — 横浜（歳入 p.3-4・歳出 p.5-6）と
   川崎（歳入 p.5-6）は**1側の表が2ページにまたがる**。範囲内のページを**連結してから**
   1つの表として読む。合計行は最終ページにしか無いので、単ページで読むと必ず
   「合計行が見つかりません」で落ちる（＝黙って欠落はしない）。`revenuePage`（単数）と併用不可。
   `samePage`（1ページに2側・南アルプス）とはちょうど逆方向の拡張。
2. **下段折返し款名が次の款へ漏れるのを修正** — 名古屋・札幌・福岡に共通する
   「中央寄せ3行折返し」（款名が金額行の**上下**に分かれ、款行の名前欄が空）:

   ```
        国有提供施設等所在      ← 上段
   4                  3,000 …   ← 款行（名前欄が空）
        市町村助成交付金        ← 下段 ★これが款5 の名前を汚染していた
   5    地方特例交付金  …
   ```

   既知の折返し（豊川・和泉）は**切れるだけ**だったが、政令市は款の間に空行が無いため
   下段断片が次の款名の頭に付く（`交付金分離課税所得割交付金` 等）。**金額は全件正しく
   Σ款=合計も通るので validate は素通りする**＝款名だけが静かに壊れる型。
   判別は「**款行の名前欄が空だった款だけを下段待ちにする**」。名前欄が非空＝行内で名前が
   完結している款は下段待ちにしないので、上段折返し（`国有提供施設等所在` + `4 市町村助成交付金 3,000`）の
   既存挙動は変わらない。**副次効果として §7 の残タスクだった豊川 款11・和泉 款10 の
   「国有提供施設等所在市町村助成交付金」が完全復元された**（既存20ソースの再 parse で
   実データの差分はこの2件のみ＝回帰0）。
3. **単位だけの行を款名断片から除外** — 横浜は列見出しと別に `千円 千円 千円` の行があり、
   これが款名に連結されて `千円千円千円市税`・`千円千円千円地方特例交付金` になっていた
   （ページ連結で2ページ目の反復分も拾い、款13 でも再発）。**`KAN_HEADER_RE` に `千円` を
   足して行ごと弾くのは不可** — 款行に単位がインラインで入る様式（`35,300,000 千円`）を
   巻き添えにする。断片側で `^(千円|百万円|円)+$` を弾く形にした。
   **これも validate を素通りする型**（金額は正しい）。**新資料は必ず款名を目視すること**。

#### 検証ゲート（設計済み・実データで成立を確認）

この資料は**自己検証が2本立つ**（「新しいデータ種別は先に検証ルールを設計する」の実践）:

1. **総コスト(A+B) = 事業費A + 人件費B** — 地域子育て支援事業の**10列すべてで厳密一致**
   （606,035 = 376,902 + 229,133 …）。**列の取り違えを算術で検出できる**のが大きい。
2. **財源内訳（国庫支出金＋市債＋その他特財＋一般財源）の和 = 事業費A**
   （112,665 + 0 + 113,163 + 151,074 = 376,902）。
3. 事業数 = **572**（`gaiyou.pdf` p.1 に明記）。達成度の内訳（2=17 / 3=462 / 4=93）も 572 に一致。

#### 予決算表は `-tsv` の座標が必須（`-layout` では列がずれる）

**トークン数で列を対応させてはいけない**。R7 の決算額は 事業費A では**空欄**だが
人件費B・総コストでは **0** が入るため、行ごとにトークン数が変わる（A=10・B=11・総コスト=11）。

```
        R4年度                    R5年度                  R6年度                R7年度
   予算額   決算額  計画事業費   予算額  決算額 計画事業費  予算額 決算額(見込) 計画事業費  予算額 決算額
A  376,902 346,321 363,697  412,100 385,438 363,697 499,986 477,565 363,697 480,414   ← 10個
B  229,133 229,133 191,642  191,642 191,642 190,959 190,959 190,959      0      0  0  ← 11個
```

**列境界はヘッダ（`予算額`/`決算額`/`計画事業費`）の x 座標から導く**（決め打ちしない。実測:
R4予算額 x=178.7 / R4決算額 x=216.5 / … / R7決算額 x=511.3 で11列）。値は各列の右端に右揃え。
**空セルは `-` トークンとして現れる**ので、欠測と 0 を区別できる。

#### 既存スキーマは流用できない（要拡張）

`projectReportFactSchema`（甲府の第2号様式用）は **`grade: /^[A-F]$/`（総合評価A〜F）と
`score`（24点満点）** を必須にしている。**川崎は評価体系が違う** — **達成度 1〜5** と
**方向性区分 Ⅰ現状のまま継続／Ⅱ改善しながら継続／Ⅲ事業規模拡大／Ⅳ事業規模縮小** の2軸で、
A〜Fも点数も無い。**甲府の体系へ丸めてはならない**（「未知の科目を黙って寄せない」と同じ話で、
評価の意味が変わる）。`grade`/`score` を nullable にし、達成度・方向性を持つ形へ拡張する。
画面の評価バッジ（A〜F）もそのままでは使えない。

#### パースで踏んだ罠（すべて「金額は正しく見えるのに静かにずれる」型）

**設計した検証ゲートが実装中に3つのバグを捕まえた。** ゲートが無ければ全部素通りしていた:

1. **年度が1つずれる** — 年度ヘッダ（`R4年度`…）は3列にまたがるセルの中央に置かれるが、
   pdftotext が返すのは**ラベル自身の箱**。「x が最も近いヘッダ」で対応させると
   R4 の計画事業費が R5 に付く（実測: R4計画 中心261 / R4年度 中心208 / R5年度 中心295）。
   → **「予算額」が各年度グループの先頭**という構造を使う。
2. **同じページに年度ヘッダが2組ある** — 予決算表（y≈250）と**成果指標の表**（y≈731）。
   全部を x でソートすると `R4,R5,R4,R6…` と混ざる。→ **最も上のヘッダ行だけ**に絞る。
   あわせて「年度が連続していること」「1年度あたり3列まで」の構造チェックを parse に入れた
   （**ゲート①②は列内の縦の整合しか見ないので、年度の対応ずれは検出できない**）。
3. **人件費B が 565件で静かに欠落** — `※` が `人件費` と `B` の**間**に入る（`人件費 ※ B` の順）
   ため `includes("人件費B")` が外れ、7/572 しか取れていなかった。→ `/人件費[※\s]*B/`。

**達成度は座標で取る（読み順で取ってはいけない）** — 選択値のセルが「達成度」ラベルの**上**に
来ることがある（選択肢の並び順が資料内で一定しない。実測: `3-3.pdf` の「魅力的な公園整備事業」で
値がラベルの13ポイント上）。選択値は半角の単独数字で、ラベルより右・選択肢（全角「１．…」）の
列より左、ラベル±25pt に入る。**ただし区の「地域課題対応事業用」様式はこの条件に当たらない**
（読み順で直後に来る）ので、座標で取れなかったときだけ読み順で拾う。

#### 収録の状況（2026-07-15・error 0 / warning 1）

**R7 の2,535事業すべてを抽出し validate まで通した**（`yokohama-jigyo-hyoka-r7`）。アプリ未接続。

- その1件は**発行元の誤植**（都筑区「区庁舎管理費」: 資料が `255,960 / 263,659 / 差引 7,609` だが
  **263,659 − 255,960 = 7,699**）。3つとも原典どおり抽出できている。**ズレが小さければ warning**
  （列を取り違えていれば桁ごと違う）として原典どおり収録する

**実装中にゲートが捕まえたもの（error 3,480 → 1）**:
1. **評価書番号の重複 2,471件** — scout の警告どおり **`評価書番号` は局内で一意でない**
   （277件中ユニーク55）。素の番号を `no` に入れると validate の重複チェックが誤爆する。
   **`(ファイル, 会計, 款, 項, 目, 評価書番号)` の複合キー**にした
2. **差引ゲート 1,009件** — **負号 `▲` が独立トークンになる**（`▲ 7,810` が空白入り）。
   数字トークンだけを見ると**符号が落ち**、差引が +7,810 になって「列を取り違えた」ように見えた。
   同じ行の直前の `▲/△` を拾って負にする。**川崎の `△` は数字と同一トークンだったので、

#### 款の見つけ方 — 列0判定＋3つの例外

**款は `pdftotext -layout` の列0始まりの行**（項・目・節は字下げされる）。歳出はこれだけで
16行=16款ちょうど（偽陽性0）・Σが本年度／前年度とも厳密一致した。**歳入は3つの例外がある**:

1. **款名が2行に折返し、款番号が単独行に落ちる**（款5・6・13 のみ）。列0判定だけだと
   **20款しか取れず Σ が 8,430,001 千円不足**する。上段＝直前の断片行、下段＝直後の金額行で復元する。
   ```
       株式等譲渡所得割
   5
       交   付  金           7,371,000    5,761,000       1,610,000
   ```
2. **款22 繰越金は本年度・前年度とも「1」千円の象徴計上**。金額を桁数で絞る実装（節の枝番
   `1 現年課税分 …` を避けるために入れたくなる）だと**静かに落ちて Σ が 1 だけ合わない**。
   款行は「款番号＋数字を含まない款名＋3金額」と形が決まっているので、そちらで絞る。
3. **末尾に廃止款の括弧書き**（`（自動車取得税交付金）`・3行に割れる・**款番号を持たない**）。
   本年度0だが**前年度1千円が歳入合計に含まれる**ので、落とすと Σ前年 が 1 だけ合わない。
   → **`BudgetLineFact.kanNo` を nullable にした**（メダリオン: 原典が振っていない番号を捏造しない）。
   差引の符号が `1△   1` と数値の**後ろ**に来る（他行と逆）が、先頭2金額しか見ないので影響しない。

いずれも「金額は正しく見えるのに Σ だけ僅かにずれる」型で、**検証ゲート（Σ款=合計）が無ければ気付けない**。
最終的に歳入25款・歳出16款で**4系統（歳入/歳出 × 本年度/前年度）すべて厳密一致**。

#### 専用パーサ hiroshima-yosansho の理由

`kofu-yosansho` の骨格（1ページ＝1側の単独表）は保たれているが、前提が2つ崩れる:

1. **款番号の列が無い**（左端は `区 分`）。→ **kanNo は全件 null**。
   行順から採番することもできるが、**款の行順は明細書の款番号と一致しない**
   （廃止款の環境性能割交付金が末尾に別掲されるため、素直に採ると款10以降が全部1つずれる）。
   採番規則を推測して当てにいくと静かに1つずれるので、**番号は持たない**（原典が振っていない）。
2. **列が10個ある**（本年度 / 前年当初 / 前年2月補正後 × 予算額・構成比 ＋ 差引増減・増減率 × 2）。
   しかも**構成比・差引欄が空欄になる行がある**ので列位置では取れない。
   → **小数点を含まないトークンだけが金額**（構成比・増減率は必ず `34.3` のように小数、
   `皆減`/`皆増` は数字を持たない）という性質で拾い、**先頭2つ＝本年度・前年当初**を取る。

見出しブロックは日本語だけの行で款名の折返し断片と見分けが付かないが、
**単位行 `億 万 千円 ％ …` が見出しの最終行**なので、最初に `％` を含む行までを構造で読み飛ばす。

#### 罠

  国有提供施設等所在／交通安全対策）。**款番号が無いので折返しの起点が無い**。
- **廃止款 環境性能割交付金**（本年度0・前年 681,000・`皆減`）。款番号を持たない。
- **象徴計上 繰越金 `1/1`**（大阪 §8e と同型）。
- 前年度は**当初と2月補正後の両方が載る**ので、`prevBasis` が資料から確定できる（札幌の骨格予算問題が無い）。

#### 款名の重複を validate の error にした（2026-07-16）

款名の折返しの取りこぼしは**金額が正しいまま款名だけが化ける**ので Σ のゲートを素通りする
（横浜「千円千円千円市税」・広島「交付金」型）。**折返しに失敗すると頭の断片が落ちて後半だけが残り、
衝突して重複になる**ので、`同じ側に同じ款名が2件以上 = error` を足した。
**実際に折返し処理を殺して再現確認済み** — `交付金` ×3・`所得割交付金` ×2 で止まる。
既存59ソースに偽陽性なし。**款番号を持たない資料ではこれが唯一の網**になる。

#### §10b 特別区で繰り返し出た罠（収録7区の実測から）

**1. 「見出しは強いほど安全」は誤り**（中央・江東の2区で実害）。パーサは**見出しに一致する行を
行ごと skip する**設計なので、ページ内のタイトル行（`歳　入`）を見出しに指定すると読み飛ばせるが、
表題（`一般会計当初予算対前年度比較`）のような強い語を見出しにすると skip が効かず
**款1 が `歳入特別区税` に静かに汚染される**（Σ は差0 のままで validate も素通り）。
- 中央は**弱い見出し**（`歳入`/`歳出`）で解決。代償として**ページ誤指定が throw しなくなる**
- 江東は歳入だけ強い見出しが要る（特会と区別するため）ので **`revenueHeaderExtra: "^歳入$"` で
  タイトル行を落とす**。歳出は見出しがタイトル行そのものなので無事＝**側で非対称**

**2. 同一 PDF に特別会計の同型表が並ぶ**（中央・目黒・江東・渋谷の4区）。見出し語が同じなので
**ページ指定を誤ると throw せず別会計を静かに収録する**。江東で実測すると国保のページでも
**Σ は差0 で通った**（8款・46,896,000）。→ **守っているのは validate の「歳入合計 = 歳出合計」
（予算は同額編成）だけ**。ページを動かしたら必ず validate を通し、総額を別資料と突合する。

**3. `kanNoless` が「静かに落ちる廃止款」を副次的に救った**（目黒 R2）。原典が**款番号欄を空にする**
廃止税目行（`自動車取得税交付金 0 / 137,800`）は、無しだと**前年度 Σ が 137,800千円 不足**するが
**validate は当年度 Σ しか見ないので error にならない**（§8 の札幌6.9億・福岡1千円と同じ型）。
岡山 §9i で入った拡張が政令市で既知だった穴を塞いだ形。**R8〜R3 では no-op**（実測）。

**4. 葛飾 R8 は「第6の折返し型」**（款番号と款名の下段が**同じ行**・金額が次行）:
```
     株式等譲渡所得割      ← 上段（款名・金額なし）
５    交付金               ← 款番号＋下段（金額なし）＝款番号の単独行ではない
                1,940,000  953,000 …   ← 金額だけの行

`pendNo`（甲府R2・大阪 §8e）にも §9j の孤児機構にも乗らず**6款が落ちる**（Σ −19,532,957千円）。
`kanNoless` が救うが**その6款は `kanNo: null` になる**（原典は 5・6・7・10・11・12 と振っている）。

**5. 款番号に 0 は無い、という不変条件をパーサに入れた**（葛飾 R2・パーサ改修）。R2 の廃止マーカーは
**`○`（U+25CB）**で、R4 の **`〇`（U+3007）** と別字（§4 の「同じ字に見えて別の文字」が**同じ自治体の

schema が落ちていた。**マーカーの字を足して回るのではなく `[1-9]\d*` で「款番号に 0 は無い」を効かせた**
（既存144ソースを再 parse して `parsedAt` 以外の差分ゼロを確認済み）。あわせて款名の掃除にも
`○` を足した（落とさないと `自動車取得税○` になり**Σ を素通りする**）。

**6. 偵察は「款名は全件クリーン」と報告してくるが信用しない**。江東は実際には**7年度すべてで款1 が

### §9j 左右2側が同一ページ、と第5の折返し型（2026-07-16・静岡 R8）

#### 左右2側が同一ページ（横並び）

静岡の総括は**歳入（左）と歳出（右）が同じページに並ぶ**:

```
１ 市 税  148,938,000  145,700,000  3,238,000  │  １ 議 会 費  1,021,222  1,026,482  △5,260
```

`-layout` は2表を1行に融合するので、**歳入だけ偶然正しく出て（先頭2列が歳入なので）歳出が1件も
取れない**（＝「款行が1件も抽出できませんでした」で throw する。静かには壊れない）。

既存のページ拡張はどれも別方向:

| 拡張 | 意味 |
|---|---|
| `revenuePages` | 複数ページを**縦に**連結（横浜・神戸・北九州） |
| `samePage` | **1ページに2側が縦積み**（南アルプス・岡山） |
| `revenueSpread` | **2ページを行順で1:1**（新潟） |
| **`revenueCropX`** | **1ページを横に切る**（静岡） |

**切り出しは `pdftotext` 自身の `-x`/`-W` に任せる**（座標を自前で組み直さない）。
静岡は A4 横（842pt）で歳入の右端が約400pt・歳出の左端が約425pt＝**ガター24〜26pt**。閾値 412 が安全。

#### 第5の折返し型 — 金額が款番号より前（上段）に来る

```
     農    林     5,152,466   4,616,851   535,615 …   ← 上段に款名前半＋金額
６                                                     ← 款番号が単独行
     水 産 業費                                         ← 下段に款名後半
```

既存の `pendNo`（甲府 R2/R3・大阪 §8e）は**「款番号 → 金額行」の順**を前提にしている。
この順では**上段が孤児として黙って捨てられ、款が丸ごと落ちる**:

- 静岡 R8 は**款6・款11 が消えて Σ が 10,976,396 千円不足**（= 5,152,466 + 5,823,930）
- さらに下段「復旧費」が次の款へ漏れて**款12 が「復旧費公債費」**になった

**Σ ゲートが止めるので静かには壊れない**が、**款名の汚染は Σ を素通りする**（もし款が落ちずに
名前だけ漏れる版面なら気づけない）。

→ 「金額はあるが款番号が無く、どの款にも結び付かない行」を**孤児として1行だけ覚えておき、
直後に款番号の単独行が来たらその款の上段だったと判定する**（構造判定・オプション不要）。
既存68ソースを再 parse して**差分が `parsedAt` だけ**であることを確認済み。

### §9h 団体コードの取り違えは Σ も画面も素通りする（2026-07-16・浜松で実害）

**浜松市を団体コード `221007`（＝静岡市のコード）で登録していた**（正: `221309`）。
本番で **浜松の1人あたり予算が約16.5%過大**（783,924人で割るべきところ静岡の672,775人で割っていた）。

**なぜ全部の網を抜けたか**:
- **Σ ゲートは款別の話**なので通る（金額は正しい）
- **derive も止まらない** — `221007` は**実在する別の自治体**なので人口レコードが見つかってしまう
- **画面にも「浜松市 … 672,775人」と「それらしく」出る**ので、目視でも見落とした（実際に見落とした）
- スカウトの提案をそのまま採り、**団体コードを総務省の実データと突合しなかった**のが原因
  （他市では突合していたのに、ここだけ抜けた）

→ derive に **「団体コードに対応する総務省の自治体名が registry の名前と一致するか」** の検査を足した。
名前で突合すれば一撃で落ちる。**わざとコードを戻して発火することを確認済み**:

```
error: hamamatsu-yosansho-r8: 団体コード 221007 は総務省R6では「静岡市」です（registry は「浜松市」）。
       コードの取り違えです — 人口が別の自治体のものになり、1人あたりが静かに狂います
```

**教訓**: 金額の Σ は「金額の網」でしかない。**結合キー（団体コード）には別の網が要る**。
「実在する別のコード」は存在チェックを素通りするので、**存在ではなく同一性を検査する**。

### §9g license 欄に「適用されない規約」を書くと逆の区分に落ちる（2026-07-16・熊本で実害）

`licenseClassOf` は **license 欄の自由文を語で判定する**（`政府標準利用規約|公共データ利用規約` → open）。
**文脈を読まない**ので、**適用されない規約の名前を説明として書くと逆の区分に落ちる**。

熊本の license 欄に事実関係として
「公共データ利用規約は BODIK カタログ限定で**本 PDF には適用されない**」と書いたところ、
**open に分類された** — **発行元が許諾していないのに「自由に使える」と画面に出すところだった**。

→ **license 欄には「適用される条件」だけを書き、経緯は registry のコメントに置く**。

なお open 判定の全14件を洗い直し、**残りは総務省（公共データ利用規約）と神戸（政府標準利用規約準拠）

### §9f 款番号の「連番」チェックは偽陽性しか生んでいなかった（2026-07-16 に撤去）

`validate` には長らく **「款番号が 1,2,3,… でなければ warning」** という規則があった。
全ソースを洗ったところ、**発火していた2件はどちらも原典どおり**だった:

- 富士河口湖 歳出 `1..10,12` … 款11（災害復旧費）が**実在しない**だけ
- 相模原 歳入 `5,10,13,16,…,90` … 原典が**5刻み前後で独自採番**している（熊本も同型）

warning は derive が「データの注意」として**市民の画面に出す**ので、これは実害だった。
**欠番なら Σ が合わない**（金額ごと落ちる）ので Σ ゲートが本命であり、番号の連番性は要らない。

→ **重複（error）と昇順の逆転（warning）だけを見る**ようにした。逆転は「行の取り違え」という
実際のバグを指す。撤去後、**全ソースで款番号の warning が0件**になり、
「データの注意」は 6自治体19件 → **5自治体18件** に減った。

### §9e 見開き2ページ型 — 款名と金額が別ページに載る（2026-07-16・新潟 R8）

### §9d 第4の折返し型 — 幾何では見分けられない（2026-07-16・仙台 R8）

### §9b Wayback が大きなファイルを MiB 境界で打ち切る（2026-07-16 に発見）

**魚拓の sha256 不一致には意味が2つあり、正反対**なので混ぜてはいけない:

- **Wayback 側の打ち切り** … 一部のクローラが大きなファイルを **MiB 境界ちょうど**で切る。
  **私たちの raw が正しく、Wayback が壊れている**。--force で再登録しても同じ上限で切られる。
- **別版** … スナップショットが古い版を指している。--force で現行版の再登録が要る。

**上限は1つではない**（最初は「5 MiB ちょうど」を定数にしたが、**同日に 1 MiB の実例が出て反証された**）:

| ソース | raw | Wayback |
|---|---|---|
| kobe-yosansho-r8 | 25,118,962 bytes | **5,242,880（5 MiB）**・pdftotext が何も出せない |
| yamaguchi-yosansho-r7 | 5,588,963 | 5,242,880（5 MiB） |
| kyoto-yosansho-r5 | 5,810,234 | 5,242,880（5 MiB） |
| kyoto-yosansho-r4 | 5,276,547 | 5,242,880（5 MiB） |
| **hamamatsu-yosansho-r5** | 3,885,492 | **1,048,576（1 MiB）** |

**単純なサイズ上限ではない**（名古屋 R8 16MB・神戸 R7 12MB・R6 11MB はいずれも一致している）ので、
クロールの条件によるものと見られる。→ **境界そのもの（1 MiB の倍数）で判定する**。
実ファイルが 1 MiB の倍数ちょうどになる確率は100万分の1程度で、かつ「raw より小さい」
「sha が不一致」も同時に満たす必要があるので誤検出しない。

台帳の不一致は**「別版」が0件・打ち切りが7件**（2026-07-16 時点）。
それまでのメッセージは打ち切りに対しても
「別版の可能性。--force で現行版を再登録してください」で、**意味が正反対の助言をしていた**。

```

**(a) 廃止税目の行を款として拾えない** — 款番号欄が数字でない（甲府 R2 `廃款` / 札幌 `△` /
福岡 `▲` / 横浜 R2・H31 は括弧書き）。`lead` 正規表現に当たらず**行ごと落ちる**。
`kanNo` は正の整数必須なので**表現する手段が無い**（schema 変更が要る）。

**(b) 前年度欄が空の款で「比較」列を前年度と誤読** — その年度に**新設**された款は前年度欄が
空欄のまま伸率欄に `皆増` と書かれる。整数列が `[当年度, 比較]` の2個になるため、
「整数列は必ず [当年度, 前年度, 比較] の順」という**パーサの中核の前提が崩れ**、
`ints[1]`（＝比較）を前年度として読む。**甲府 R2 の法人事業税交付金は前年 190,691 と
表示されていた（正: 0 からの皆増）** — R2 に新設された交付金なので前年は存在しない。

**この2つは当年度 Σ を汚さないので、当年度だけを検査していた validate を素通りしていた。**
甲府 R2 では **(a) −76,900 と (b) +190,691 が符号違いで部分的に相殺**し、
前年度Σのズレが +113,791 に圧縮されていた（**額が一致すれば完全に相殺して完全に消える**）。

**対策2点（2026-07-15）**:
1. **`validate` に「款の前年度の和 = 前年度合計」を追加**（warning）。(a) は表現できない既知の
   欠落なので error にはせず、人が「既知の欠落」と「新しいバグ」を判別できる文面にした。
   全25ソースに掛けて**挙がったのは3件だけ・すべて (a) の既知欠落**（甲府R2 −76,900 /
   札幌R8 −694,000 / 福岡R8 −1）＝**誤検知ゼロ**。
2. **`皆増` / `皆減` を「相手側のセルが 0」の記号として使う**（列位置を推測しない）。
   `皆増` → `prevAmount = 0`、`皆減` → `amount = 0`。札幌 R8 のように 0 が明記されている
   様式（`0 0.0 694,000 0.1 △694,000 皆減`）とも矛盾しない。これで (b) は解消した
   （既存25ソースで実データの差分は**甲府 R2 の1数値のみ**＝回帰0）。

**(a) は未解決のまま**（前年度Σが僅かに不足する。甲府R2 0.1% / 札幌R8 0.05% / 福岡R8 1千円）。
検証ゲートが毎回 warning を出すので、もう静かではない。

**この罠は税制改正のたびに再発する**（款の新設・廃止が起きる）。**新しい年度・自治体を足したら
warning の有無を必ず読むこと。**
